# Outreach Plan — v0.9.0

## Purpose

This plan prevents chaotic public exposure.

The goal is not broad publicity.

The goal is controlled hostile review.

## Sequence

### Step 1 — Internal readiness check

Confirm these files exist and pass checks:

    PAPER_SHORT.md
    TECHNICAL_SUMMARY.md
    REVIEWER_CHECKLIST.md
    REVIEW_QUESTIONS.md
    ERROR_ATLAS.md
    COMPARATIVE_ANALYSIS.md
    DIFFERENCE_MATRIX.md
    OPERATIONAL_AUDIT.md

### Step 2 — Select five reviewer classes

Start with:

    AI safety / AI governance
    model risk management
    systems safety / STPA
    philosophy of science / epistemology
    formal methods / logic

### Step 3 — Send only narrow request

Do not ask for recognition.

Ask for:

    strongest existing equivalent
    strongest objection
    weakest definition
    counterexample
    operational usefulness

### Step 4 — Collect objections in GitHub issues

Every serious objection should become an issue.

Each issue should be classified as:

    counterexample
    definition weakness
    redundancy claim
    case atlas objection
    operational use case

### Step 5 — Build v1.0.0 from objections

v1.0.0 should not be a celebration release.

It should be a review-response release.

It should include:

    external objections
    accepted corrections
    rejected objections with reasons
    revised definitions
    stronger case scoring
    narrowed claims if required

## Outreach Rule

Do not lead with the largest philosophical claim.

Lead with:

    local correctness does not imply valid use.

Then let reviewers discover the scope.

## What To Avoid

Avoid public posts before at least one hostile technical review.

Avoid broad claims without external criticism.

Avoid framing the work as complete.

Avoid seeking approval instead of attack surface.

## Success Criterion

The first success is not praise.

The first success is a precise objection strong enough to improve the framework.
